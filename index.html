Hypothesis & analysis approach

The question we are trying to answer and why it is hard to prove.

The hypothesis
Publishing a funnel during the trial period independently causes higher trial-to-paid conversion — not because publishers are more engaged users overall, but because the act of getting a funnel live creates real value that drives the decision to pay.
The challenge — why this is hard to prove
The engagement confounder
What the raw data shows
Publishers convert at 72% vs non-publishers at 48% — a massive +24pp gap. But this is misleading because publishers are also running more workflows, sending more emails, creating more contacts. They're just more engaged users across the board.
What we need to know
Is the higher conversion because of the funnel publish itself? Or would these companies have converted anyway because they're generally more committed users? Answering this requires isolating the funnel effect from everything else.
The answer — from 4 independent methods
Publishing a funnel during trial independently lifts conversion by ~2–4 percentage points after controlling for plan tier, niche, engagement, capture call, trial length, client size, and 9 other product activations.

WITH trials AS (
  SELECT
    CUSTOMER_METADATA_HL_COMPANY_ID AS company_id,
    MIN(CONVERTED_TRIAL_START_DATE::DATE) AS trial_start,
    MAX(CONVERTED_TRIAL_END_DATE::DATE)   AS trial_end,
    MAX(CASE WHEN CM_CUSTOMER_STATUS != 'Lead' THEN 1 ELSE 0 END) AS converted
  FROM PROD_HIGHLEVEL_DWH.MOZART.NEW_TRIALS
  WHERE CONVERTED_TRIAL_START_DATE >= '2026-04-01'
    AND   CONVERTED_TRIAL_START_DATE  < '2026-05-01'
    AND   CONVERTED_TRIAL_END_DATE IS NOT NULL
    AND   CUSTOMER_METADATA_HL_COMPANY_ID IS NOT NULL
    AND   COUNTRY_NAME = 'United States of America'
  GROUP BY 1
),
company_locations AS (
  SELECT DISTINCT COMPANY_ID, LOCATION_ID
  FROM HL_PRODUCT_DB.PRODUCT_ANALYTICS.ACCOUNT_DAILY_METRICS_REPORT
  WHERE COMPANY_ID IS NOT NULL AND LOCATION_ID IS NOT NULL
),
loc_funnel AS (
  SELECT cl.COMPANY_ID,
    MAX(CASE WHEN fm.CREATED_AT::DATE BETWEEN t.trial_start AND t.trial_end THEN 1 ELSE 0 END) AS created_funnel,
    MAX(CASE WHEN fm.DOMAIN_ATTACHED_AT::DATE BETWEEN t.trial_start AND t.trial_end THEN 1 ELSE 0 END) AS published_funnel
  FROM company_locations cl
  JOIN trials t ON cl.COMPANY_ID = t.company_id
  JOIN HL_PRODUCT_DB.PRODUCT_LEADGEN.FUNNEL_METAS fm ON fm.LOCATION_ID = cl.LOCATION_ID
  WHERE LOWER(fm.FUNNEL_TYPE) IN ('funnel','website')
    AND fm.IS_DELETED = FALSE
    AND fm.SOURCE_TYPE != 'snapshot'
  GROUP BY cl.COMPANY_ID
)
SELECT
  CASE
    WHEN COALESCE(lf.published_funnel,0) = 1 THEN 'a) Published'
    WHEN COALESCE(lf.created_funnel,0)  = 1 THEN 'b) Created, not published'
    ELSE                                          'c) No funnel'
  END AS bucket,
  COUNT(*) AS companies,
  ROUND(AVG(t.converted) * 100, 2) AS conversion_rate_pct
FROM trials t
LEFT JOIN loc_funnel lf ON t.company_id = lf.COMPANY_ID
GROUP BY 1
ORDER BY 1

The approach

4 methods to isolate the funnel effect

If all four methods converge on the same answer despite using completely different statistical approaches, the finding is credible.

Why this analysis: The confounder check confirmed publishers are more active across every other product. So the raw +25pp gap includes both the funnel effect and the engagement effect. We need methods that strip away the engagement difference and isolate what funnel publishing alone contributes to conversion.
Each method removes confounders differently — click any method to jump to its section.
All four methods — the converging answer
Bottom line: All four methods agree on direction — publishing a funnel during trial independently lifts conversion. The controlled estimates (PSM, logistic regression) converge at +2–4pp. The funnel-only cohort shows a larger gap (+23–35pp) because it compares against a lower-intent control group. Creating without publishing shows zero independent lift across every method and every cohort. The value is entirely in getting it live.
Isolating the effect · Method 1

Funnel-only users

This removes the engagement confounder by design.

Why this analysis: If we restrict to companies that used GHL exclusively for funnels during trial — zero workflows, contacts, emails, SMS, opportunities, forms, calls, or calendar events — then the engagement confounder is removed by construction. Everyone has equally zero other-product activity. Any conversion difference between publishers and non-publishers must come from publishing itself.
Why this matters: Publishers convert higher partly because they're more engaged overall. By restricting to companies with zero other-product activity, we strip that explanation away. Whatever gap remains is attributable to funnel publishing itself — not general engagement.

Stratified SQL analysis

Feb–Apr 2026 · US only.

Why this analysis: Instead of removing engagement entirely (like the funnel-only cohort), we compare publishers vs non-publishers within identical groups — same plan tier, same niche, same engagement level, same onboarding experience. Within each group, the only difference is whether they published. We then average the results across all 112 groups. This is an "apples to apples" comparison.

Snowflake SQL — stratified weighted comparison
WITH stratified AS (
  SELECT *,
    CONCAT_WS('|',
      plan_id, COALESCE(niche,'Unknown'), primary_use,
      client_size, trial_experience,
        CASE WHEN engagement_score <= 1 THEN 'low'
           WHEN engagement_score <= 4 THEN 'medium'
           ELSE 'high' END
    ) AS stratum
  FROM features
),
stratum_rates AS (
  SELECT stratum,
    SUM(CASE WHEN bucket='A' THEN 1 ELSE 0 END) AS n_A,
    SUM(CASE WHEN bucket='B' THEN 1 ELSE 0 END) AS n_B,
    AVG(CASE WHEN bucket='A' THEN converted END) AS conv_A,
    AVG(CASE WHEN bucket='B' THEN converted END) AS conv_B
  FROM stratified
  GROUP BY 1
  HAVING n_A >= 3 AND n_B >= 3
)
-- Mantel-Haenszel weighted average
SELECT
  SUM((conv_A - conv_B) * (n_A * n_B) / NULLIF(n_A + n_B, 0)) /
  NULLIF(SUM((n_A * n_B) / NULLIF(n_A + n_B, 0)), 0) AS weighted_att_A_vs_B
FROM stratum_rates
Isolating the effect · Method 3

Propensity score matching

Robustness checked across Feb, Mar, and Apr cohorts.

Why this analysis: For each company that published a funnel, we find its "statistical twin" — a non-publisher that looks nearly identical on 15 variables (plan, niche, engagement, trial experience, and more). Then we compare conversion only within these matched pairs. If the publisher still converts higher than its twin, the effect is independent of the matched variables.

# 2. 1:1 nearest-neighbor matching with caliper 0.05

# 3. ATT = average conversion difference in matched pairs

Isolating the effect · Method 4

Logistic regression model

Trained on Feb + Mar 2026 (pooled), validated on April 2026 holdout.

Why this analysis: The previous methods (funnel-only, stratified, PSM) each removed confounders differently but required grouping or matching. Logistic regression takes a different approach — it holds all 16 factors constant simultaneously in a single model and directly reads the isolated funnel publish effect as a weight. Additionally, by training on Feb+Mar and predicting April (data the model has never seen), we can validate whether the effect is stable and not just a one-month anomaly.

How to read this: The model holds all 16 factors constant — plan tier, niche, trial length, engagement, and the 9 other products — and isolates the effect of publishing alone. This +4.3pp aligns with the range found by propensity score matching (+2–4pp) and stratified SQL (+7.8pp). Four different methods, same conclusion: publishing independently lifts conversion by roughly 2–4 percentage points.
How the model weight (0.170) translates to +4.3pp
The weight is in log-odds units — here is the exact conversion to percentage points
The logistic regression assigns each variable a weight. The weight for funnel_published is 0.170. This means: when a company publishes a funnel, 0.170 is added to its conversion score, with all 16 other factors held constant. The score is in log-odds units — an abstract scale that needs one more step to become a readable percentage.

WITH funnels_published AS (
  SELECT cl.COMPANY_ID,
    COUNT(DISTINCT fm.FUNNEL_ID) AS num_published

Time to publish within trial

Feb, Mar, Apr 2026 · US only · non-snapshot funnels only.

Why this analysis: Among companies that published a funnel, does it matter when during the trial they published? If earlier publishers convert higher, the implication for product design is clear: push users to publish as fast as possible. If there's a sweet spot, we should optimize for that timing.
⚠ Unexpected finding — same-day publishers convert LOWEST
Companies that publish on day 0 (same day they start trial) convert at only 64–70% — lower than those who publish on days 4–14 (78–84%). The sweet spot is 8–14 days into the trial, not day 1. This challenges the assumption that "faster = better" and suggests users need time to explore before publishing creates real value.

How to interpret this
Why same-day is lowest
Same-day publishers published within hours of starting their trial — before deeply exploring the platform. This speed may reflect lower deliberate intent: they set something up quickly but may not have invested enough in understanding its value, which could explain the lower conversion rate.
Why 8–14 days is the sweet spot
Companies that publish 8–14 days in have explored the platform, understood its value, and made a deliberate decision to build and publish a funnel. The time investment signals genuine intent. They also still have 6–22 days of trial left to experience the funnel working — long enough to see leads come in and decide to pay. Late publishers (15+) may not have that runway.
Implication for onboarding design: Consider a nudge at days 3–5 ("your trial is going well — ready to publish your first funnel?") that guides users toward the 8–14 day publish window where conversion is highest.
Snowflake SQL — time to publish analysis
-- First publish date per company during trial
WITH first_publish AS (
  SELECT
    cl.COMPANY_ID,
    MIN(fm.DOMAIN_ATTACHED_AT::DATE) AS first_publish_date,
    DATEDIFF('day', t.trial_start, MIN(fm.DOMAIN_ATTACHED_AT::DATE)) AS days_to_publish
  FROM company_locations cl
  JOIN trials t ON cl.COMPANY_ID = t.company_id
  JOIN HL_PRODUCT_DB.PRODUCT_LEADGEN.FUNNEL_METAS fm ON fm.LOCATION_ID = cl.LOCATION_ID
  WHERE LOWER(fm.FUNNEL_TYPE) IN ('funnel','website')
    AND fm.IS_DELETED = FALSE
    AND fm.SOURCE_TYPE != 'snapshot'
    AND fm.DOMAIN_ATTACHED_AT::DATE BETWEEN t.trial_start AND t.trial_end
  GROUP BY cl.COMPANY_ID, t.trial_start, t.trial_end
)
SELECT
  CASE
    WHEN days_to_publish = 0   THEN '0 — Same day'
    WHEN days_to_publish <= 3  THEN '1–3 days'
    WHEN days_to_publish <= 7  THEN '4–7 days'
    WHEN days_to_publish <= 14 THEN '8–14 days'
    ELSE                           '15+ days'
  END AS days_to_publish_bucket,
  COUNT(*) AS companies,
  ROUND(AVG(converted)*100,2) AS conversion_rate
FROM first_publish fp
JOIN trials t ON fp.COMPANY_ID = t.company_id
GROUP BY 1 ORDER BY MIN(days_to_publish)

Snowflake SQL
SELECT
  t.business_category,
  COUNT(*) AS total_companies,
  ROUND(SUM(CASE WHEN bucket='published'            THEN 1 ELSE 0 END)*100.0/COUNT(*),1) AS pct_published,
  ROUND(SUM(CASE WHEN bucket='created_not_published' THEN 1 ELSE 0 END)*100.0/COUNT(*),1) AS pct_created_not_pub,
  ROUND(SUM(CASE WHEN bucket='no_funnel'             THEN 1 ELSE 0 END)*100.0/COUNT(*),1) AS pct_no_funnel,
  ROUND(AVG(CASE WHEN bucket='published' THEN converted END)*100,1) AS conv_if_published
FROM company_bucket
WHERE business_category NOT IN ('Unknown', 'Other')
GROUP BY 1
HAVING COUNT(*) >= 50
ORDER BY pct_published DESC
